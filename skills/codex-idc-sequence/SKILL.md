# Codex IDC Sequence

## Runtime Contract

This is the Codex adapter for the Claude Teams `idc-sequence` playbook. Do not
edit or run `../../agents/idc-sequence.md` (relative to this skill directory inside
the idc-workflow plugin) from Codex. Use it only as the
donor contract for authority boundaries.

Run as the parent Codex session. Use Codex subagents for bounded plan reading,
matrix synthesis, tracker editing, and review. The parent owns every TRACKER
edit. Per `architecture.md §Cross-runtime substrate model option 2 (inline-read,
recommended short-term)`, this skill **inline-reads** the corresponding Claude
roleplayer agent files at run time and uses each file body as the orientation
prompt for the Codex subagent that performs that role's workflow. The shared
roleplayer-substrate model means the contracts are byte-compatible across
runtimes.

## Authority

Allowed writes:

- `TRACKER.md` — wave ordering and status only, by polished pillar-derived work.
- `docs/workflow/pillar-matrices/<phase-tag>-matrix.yaml` — polished rough-matrix
  substrate (Phase 2 only). Three sibling files (`<phase-tag>-dag.mmd`,
  `<phase-tag>-parallel-safety.md`, `<phase-tag>-waves.md`) regenerate via
  `python docs/workflow/scripts/pillar_matrix.py --view={dag,parallel-safety,waves}`
  and ship in the same PR (AUTOGENERATED-sibling contract).
- Optional wave handoff under `docs/workflow/handoffs/waves/<YYYY-MM-DD-HHMM>-<tag>.md`.
- Run-audit under `docs/workflow/audits/<YYYY-MM-DD-HHMM>-sequence-run-audit.md`
  (CS-1 `idc-skill-run-audit` per the uniform IDC role-run-audit shape). <!-- lint-allow: dangling ref, tracked in docs/dev/known-debts.md -->
- Code-review under `docs/workflow/code-reviews/<YYYY-MM-DD>-tracker-edit-review.md`
  (only when QR-5 review surfaces findings).
- Scratch files under `/tmp/idc-sequence/<run-id>/`.

Forbidden writes:

- PRD, architecture spec, master plan, subphase plans, pillar plans, source code,
  tests.
- New phase, subphase, pillar, or implementation scope.
- Non-`(idle)` `Currently building:` lane pointers (Build's authority).
- Bookend-close commits (Build's authority — Sequence emits bookend-open during
  admission only).
- `CLAUDE.md`, `AGENTS.md`, or subdir rule files (route through Ripple).

## Procedure

The Codex parent runs the same five-phase workflow as the Claude `idc-sequence`
parent, but uses Codex subagents for the per-phase work. For each role below,
the parent **inline-reads** the named Claude roleplayer agent file (via Codex's
`Read` tool; agent/skill paths below are relative to this skill directory inside the idc-workflow plugin) and passes the file body as the Codex subagent's orientation
prompt. Subagents are bounded read+write per their role's authority boundary.

### Phase 0 — Preflight

#### Worktree isolation (MANDATORY)

This skill must be running in an isolated worktree branched off `main`, not directly on `main`. The mandate matches the Claude IDC roles per `WORKFLOW.md §9.2 — Worktree mandate per role`; running any IDC role on `main` directly is forbidden so parallel sessions stay isolated.

1. **Self-check.** `git branch --show-current` MUST NOT return `main` or `master`. If it does, halt and either:
   - Instruct the operator to invoke this skill from a non-`main` starting branch, OR
   - Auto-create a worktree:
     ```bash
     git worktree add -b codex-sequence/<slug> .claude/worktrees/codex-sequence-<slug>
     cd .claude/worktrees/codex-sequence-<slug>
     ```
   `cd` into the worktree immediately.
2. **Slug derivation.** Deep admission: the wave / admission tag (kebab-case). Janitor mode: `janitor-<YYYY-MM-DD-HHMM>`.
3. **Record at session start** — capture the branch + worktree path in `<scratch>/codex-cleanup-manifest.md` per §Branch and worktree cleanup below.
4. **Cleanup at session close** uses Variant A of `WORKFLOW.md §9.2` (filesystem backend with PR landed) or worktree-reap-without-PR (github backend; TRACKER state already landed via API) — see §Branch and worktree cleanup below.

Branch prefix is `codex-sequence/<slug>`. Worktree path is `.claude/worktrees/codex-sequence-<slug>/`.

#### Preflight steps

1. Read `TRACKER.md` header, `AGENTS.md`, root `CLAUDE.md`, master plan section
   for the pillars in scope.
2. Verify the input is admissible material: polished pillar plan paths, phase
   plan paths, or handoff paths supplied by the operator. For `--janitor`, no
   input is required (discovery from TRACKER + active plan).
3. Confirm `git rev-parse --show-toplevel` resolves and `TRACKER.md` exists.

### Phase 1 — Repo truth ingestion (parallel)

Dispatch two Codex subagents in parallel, each oriented by inline-reading the
named Claude roleplayer agent file:

- **`pillar-plan-ingester`** — orient via `Read ../../agents/idc-sequence.md (Phase 1 ingestion folded inline; PR-5)`
  and follow its 5-phase workflow as a Codex subagent. Mode parameter ∈
  `{deep, janitor}`. Emits `<scratch_dir>/work-units.yaml`.
- **`repo-truth-reconciler`** — orient via `Read ../../agents/idc-sequence.md (Phase 1 git-log reconciliation folded inline; PR-5)`
  and follow its 5-phase workflow. Probes git evidence per work unit; invokes
  CS-4 `idc-skill-ripple-verdict` per drift-flagged unit. Emits
  `<scratch_dir>/repo-truth-report.yaml`.

If CS-4 returns `ripple-required` for any drift-flagged unit, follow §Ripple
trigger before proceeding.

### Phase 2 — Matrix consolidation (deep admission only)

Dispatch one Codex subagent oriented via
`Read ../../agents/idc-sequence.md (Phase 2 matrix synthesis folded inline; PR-5)`. Follow its
10-step workflow as a Codex subagent. Includes:

- Read TRACKER `## Implementation Wave Queue` partition.
- Drop completed pillars; lock active pillars byte-for-byte from prior matrix.
- Invoke WM-3 `idc-skill-pillar-matrix-synth`, WM-5
  `idc-skill-pillar-matrix-synth`, WM-4
  `idc-skill-pillar-matrix-synth` (in that order) to synthesize the three
  view fragments for pending pillars.
- Consolidate to `docs/workflow/pillar-matrices/<phase-tag>-matrix.yaml`.
- Regenerate the 3 sibling AUTOGENERATED files via
  `python docs/workflow/scripts/pillar_matrix.py --view={dag,parallel-safety,waves}`.
- Emit initial `Currently building: (idle)` lane block at admit-time only.
- **Defer fence verification (`uv run pytest tests/test_arch_pillar_matrix.py -q`)
  to a bounded Codex shell-tool invocation** (not the synthesizer subagent's own
  context — Q-seq-1 verbatim, codex-runtime equivalent of Claude's read-only
  Task subagent dispatch).

The re-synthesis discipline + AUTOGENERATED-sibling contract + initial-lane
emission rule are codex-runtime-agnostic; the file body owns the contract.

Skip Phase 2 in `--janitor` mode unless a re-synthesis trigger fires.

### Phase 3 — TRACKER edit

Dispatch one Codex subagent oriented via
`Read ../../agents/idc-sequence.md (Phase 3 TRACKER edit folded inline; PR-5)`. Mode parameter ∈
`{deep, janitor, fix-loop}`. Subagent invokes:

- QS-1 `idc-skill-tracker-wave-queue-edit` for the actual TRACKER body <!-- lint-allow: dangling ref, tracked in docs/dev/known-debts.md -->
  emission (preserves bootstrap header byte-for-byte; emits 3-tier shape;
  refuses bookend-close).
- CS-4 `idc-skill-ripple-verdict` to confirm `verdict: tracker-only`.
- Bounded shell tool to run `uv run pytest tests/test_arch_tracker_bootstrap.py -q`.

The proposed TRACKER body is staged at `<scratch_dir>/proposed-tracker.md`. The
Codex parent commits after Phase 4 review clears.

### Phase 4 — Review + landing

Dispatch one Codex subagent oriented via
`Read ../../agents/idc-sequence.md (Phase 4 review folded inline; PR-5)`. Subagent invokes
QS-2 `idc-skill-plan-review`; on Blocker/Major findings,
dispatches the Phase 3 subagent again with `mode: fix-loop` (loop ceiling 3).

Final reviewer report only emitted at
`docs/workflow/code-reviews/<YYYY-MM-DD>-tracker-edit-review.md` if any finding
landed.

After review PASS, the Codex parent:

1. Copies `<scratch_dir>/proposed-tracker.md` → `TRACKER.md`.
2. Stages + commits with the appropriate `tracker:` message (bookend-open if
   admission triggers a fresh stage; cleanup form for janitor / fix-loop).
3. Pushes (per `docs/workflow/CLAUDE.md §Auto-pushable bookend commits` —
   commits touching only `TRACKER.md` are auto-pushable).
4. Opens a PR titled
   `sequence: TRACKER admit <slug> (wave-N)` (deep) or
   `sequence: tracker janitor — <date>` (janitor). PR body cites every
   plan-derived unit's source plan path, governance-auditor verdict, ripple
   obligations.

### Phase 5 — Closeout

Drop the role-run-audit at
`docs/workflow/audits/<YYYY-MM-DD-HHMM>-sequence-run-audit.md` via CS-1
`idc-skill-run-audit` (or via CR-5 `idc-role-closeout-author --role sequence` <!-- lint-allow: dangling ref, tracked in docs/dev/known-debts.md -->
when the Codex sibling integrates the closeout-author roleplayer in P1).

Drop the handoff at
`docs/workflow/handoffs/waves/<YYYY-MM-DD-HHMM>-<tag>.md` via CS-2
`idc-skill-role-handoff` AFTER the audit lands. The handoff frontmatter's <!-- lint-allow: dangling ref, tracked in docs/dev/known-debts.md -->
`next_role` and `auto_advance_eligible` MUST agree with the audit's `verdict`
and `halt_reason`; disagreement = halt + audit.

Path discipline: `handoffs/` (no hyphen). Hyphenated `hand-offs/` is the
banned legacy archaic form.

## Ripple trigger

If CS-4 returns `ripple-required` (sequencing exposes a real canonical
correction, not just tracker reordering), STOP TRACKER edits for the affected
unit(s). Dispatch a Codex subagent oriented via
`Read ../../agents/idc-sequence.md (Ripple-trigger step folded inline; PR-5)` (CR-8). The subagent stages a
Ripple change-order proposal at `<scratch_dir>/ripple-proposal-<slug>.md` and
returns the 4-value Ripple verdict via RS-2 `idc-skill-ripple-verdict`. If
`MAJOR_GATED` → halt with operator gate. Park affected units; continue with
non-affected units. Operator routes the proposal to `idc-ripple` for canonical
drafting.

## Output Requirements

TRACKER edits must show:

- Existing pillar trace key for each unit.
- Wave order and dependency rationale (derived from `<phase-tag>-matrix.yaml`,
  not invented).
- Blocking operator gates left open.
- Completed/deferred/rescoped stale items clearly labeled.
- No invented scope.

Phase 2 PRs (deep admission) must ship `<phase-tag>-matrix.yaml` plus all 3
sibling AUTOGENERATED views in the same commit (`<phase-tag>-dag.mmd`,
`<phase-tag>-parallel-safety.md`, `<phase-tag>-waves.md`).

Closeout PRs must include the run-audit + handoff with consistent
`verdict`/`halt_reason` ↔ `next_role`/`auto_advance_eligible` cross-check.

## Goal parity (Claude-side)

Sequence is the courier between Plan's `exit_criteria` authorship and Build's `/goal` consumption. To keep the recipe surface adapter-symmetric:
- Codex Sequence emits the same `## Per-pillar /goal recipes` block in wave handoffs (`docs/workflow/handoffs/waves/<...>.md`) as the Claude-side `idc-sequence.md` schema.
- One subsection per admitted pillar: verbatim `exit_criteria` quote + pre-assembled `writer-recipe:` + `fixer-recipe:` lines (same template strings as Claude-side).
- On cold dispatch (no wave handoff), Codex Build's bootstrap reads pillar plans directly — equivalent fallback to the Claude-side path.

Canonical shape: `../../agents/idc-sequence.md` §wave-handoff schema. Codex Sequence MUST emit identical block structure so a Codex-side admission feeds a Claude-side Build without re-derivation (and vice versa).

## Branch and worktree cleanup

Codex lacks `TeamDelete` semantics, so worktree + branch cleanup is the parent's responsibility — but parents have historically left branches dangling on the remote (see `docs/workflow/audits/2026-05-14-codex-orphan-branch-sweep-audit.md` for the cleanup of 17+ such branches). Every run of this skill MUST follow the cleanup discipline below.

1. **Record at session start.** Capture the branch name + worktree path at session start in `<scratch>/codex-cleanup-manifest.md`:
   ```markdown
   # Codex cleanup manifest — codex-idc-sequence
   - branch: codex-sequence/<slug>
   - worktree_path: .claude/worktrees/codex-sequence-<slug>/
   - main_checkout: <governed-repo>
   - pushed_at: <timestamp-if-pushed>
   ```
2. **On normal completion** — invoke the worktree-merge single-shot pattern verbatim per `WORKFLOW.md §9.2`:
   ```bash
   cd "$MAIN" && \
     gh pr merge "$PR_NUM" --squash --delete-branch && \
     git pull --ff-only && \
     git worktree remove "$WT_PATH" && \
     git worktree prune && \
     git branch -D "$BRANCH"
   ```
3. **On abort, crash, or operator stop** — Codex surfaces the manifest path + cleanup-required signal in its SUCCESS / BLOCKED telegram. The operator (not Codex) runs the cleanup manually using the manifest:
   ```bash
   cd "$MAIN" && \
     git worktree remove "$WT_PATH" && \
     git worktree prune && \
     git branch -D "$BRANCH" && \
     git push origin --delete "$BRANCH"  # only if pushed
   ```
4. **Telegram requirement.** Every SUCCESS / BLOCKED telegram from Codex MUST include `cleanup_manifest_path: <scratch>/codex-cleanup-manifest.md` AND `cleanup_required: true|false` (`false` only if the worktree-merge single-shot pattern completed in step 2). This is the **declared parent's responsibility** — operator silence on cleanup is interpreted as "Codex completed it" only when `cleanup_required: false`.
