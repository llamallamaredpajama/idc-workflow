## Findings

### Minor 1 — smoke does not lock the production `coms_net_send` seam (test fidelity)
`phase8-coms-net-probe.ts` imports `evaluateComsNetSendForRole` and **hand-rolls** the gate
(`guardedSend`) rather than invoking the actual `coms_net_send` tool handler. The decision
function (the substantive policy) is genuinely exercised end-to-end against the booted hub, and
the production gate at `coms-net.ts:1447-1461` is correct **by inspection** — but the smoke would
stay GREEN even if a future change deleted the seam's ACL call or the `prompt_blocked` emission.
The ~15 lines of handler glue (role resolution + gate + audit log) are uncovered. Wiring is
correct today; this is a regression-locking gap, not a current defect. The probe header is honest
about driving "the SAME decision function the production seam uses."
*Suggestion (follow-up, non-blocking): assert a `coms-net-log` `prompt_blocked` entry is produced
by the real send path, or factor the handler so the test drives the same entrypoint.*

### Nit 2 — `prompt_blocked` audit line unasserted
Subsumed by Minor 1: the Done-When/brief emphasize a logged denial, but the smoke never inspects
the `coms-net-log` channel. Code emits it correctly (`coms-net.ts:1451-1458`).

### Nit 3 — env-first role identity is launcher-trusted
`readIdcRoleForComsNet` reads `PI_IDC_ROLE` first. A process that sets its own env to an upstream
role (e.g. `think`) would see everything as downstream and route around the wall. This matches the
existing harness trust model — the launcher (`idc-pi`) provisions identity; the glass wall guards
an LLM resident's tool-calls, not a process that fully controls its own environment — and is
consistent with prior `PI_IDC_ROLE` usage. Acceptable for this unit; launcher/orchestration
provisioning is B2's concern. Noted for the record.

### Nit 4 — doctor Check 6 precedence prose
When BOTH Bun and Pi are absent, `install-pi.sh --check` exits non-zero (Bun is a hard prereq)
while Pi-absent is meant to map to SKIP. The bullet list ("Pi agent ABSENT → SKIP" vs "probe
non-zero → fail-closed FAIL") can read as conflicting in that one edge case. Common cases
(Pi absent + Bun present → `--check` exits 0 → SKIP) are unambiguous. LLM-interpreted prose;
a one-line precedence note would remove the ambiguity.
